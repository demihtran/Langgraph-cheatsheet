LG

LangGraph Overview

Low-level orchestration for stateful agents
Concept Framework for building agents and workflows as graphs with state, control flow, and persistence.
Use when Multi-step agents, human-in-the-loop flows, long-running workflows, and multi-agent systems.
Install pip install -U langgraph
Core APIs Graph API, Functional API, prebuilt agents, LangGraph Platform.
pip install -U langgraph langchain
pip install -U langchain-openai

# Main idea:
# StateGraph defines state.
# Nodes update state.
# Edges decide next node.
# Checkpointers persist execution.

Best: Use LangGraph when you need clearer state and control flow than a simple chain or agent.

Mistake: Build a custom graph when LangGraph's create_react_agent already fits.

AG

Build An Agent

Prebuilt ReAct agent first, custom graph when control flow matters
from langchain_core.tools import tool
from langchain_openai import ChatOpenAI
from langgraph.checkpoint.memory import InMemorySaver
from langgraph.prebuilt import create_react_agent

@tool
def search_docs(query: str) -> str:
    """Use to search internal product documentation."""
    return "Relevant product documentation snippet."

@tool
def create_ticket(title: str, severity: str) -> str:
    """Use only after the user confirms ticket creation."""
    return f"ticket created: {title} / {severity}"

model = ChatOpenAI(model="gpt-4.1-mini", temperature=0)
checkpointer = InMemorySaver()

agent = create_react_agent(
    model=model,
    tools=[search_docs, create_ticket],
    checkpointer=checkpointer,
    prompt=(
        "You are a support agent. Search docs before answering. "
        "Ask for confirmation before creating tickets."
    ),
)

config = {"configurable": {"thread_id": "support-42"}}

async def run_agent():
    result = await agent.ainvoke(
        {"messages": [{"role": "user", "content": "How do I fix API timeout errors?"}]},
        config,
    )

Start here: create_react_agent already gives you model calls, tool execution, message state, streaming, and checkpointing.

Go custom when: you need approval gates, deterministic routing, multi-agent teams, map-reduce, retries, or state editing.

LOOP

Custom Agent Loop (Async)

Model node -> tools node -> route until no tool calls
from typing import Literal
from langchain_core.tools import tool
from langchain_openai import ChatOpenAI
from langgraph.graph import MessagesState, StateGraph, START, END
from langgraph.prebuilt import ToolNode

@tool
def lookup_order(order_id: str) -> str:
    """Return order status."""
    return "shipped"

tools = [lookup_order]
model = ChatOpenAI(model="gpt-4.1-mini", temperature=0).bind_tools(tools)

# Async node function
async def call_model(state: MessagesState):
    response = await model.ainvoke(state["messages"])
    return {"messages": [response]}

def route_tools(state: MessagesState) -> Literal["tools", "__end__"]:
    last = state["messages"][-1]
    return "tools" if getattr(last, "tool_calls", None) else END

builder = StateGraph(MessagesState)
builder.add_node("agent", call_model)
builder.add_node("tools", ToolNode(tools))
builder.add_edge(START, "agent")
builder.add_conditional_edges("agent", route_tools)
builder.add_edge("tools", "agent")

graph = builder.compile()

Why this matters: this is the mental model behind tool-calling agents: call model, execute tools, append tool messages, call model again.

Do not hide risk: add interrupt() before side-effect tools such as payments, deletes, emails, or ticket creation.

SG

StateGraph

Main graph builder for stateful workflows
Concept Builder defines state schema, nodes, and edges, then compiles them into a runnable graph.
Use when Workflows with shared state and multiple coordinated steps.
Import from langgraph.graph import StateGraph, START, END
Methods add_node, add_edge, add_conditional_edges, compile
from typing import TypedDict
from langgraph.graph import StateGraph, START, END

class State(TypedDict):
    topic: str
    joke: str

def write_joke(state: State):
    return {"joke": f"Short joke about {state['topic']}"}

builder = StateGraph(State)
builder.add_node("write_joke", write_joke)
builder.add_edge(START, "write_joke")
builder.add_edge("write_joke", END)

graph = builder.compile()
result = graph.invoke({"topic": "graphs"})

Best: Type state clearly with TypedDict or Pydantic; nodes should return partial state updates.

Mistake: Mutate state directly or forget to compile the graph before invoking it.

ST

State & Reducers

Shared data and merge behavior
Concept State is the data each graph node reads and writes; reducers decide how updates merge.
Use when Append messages, aggregate results, parallel branches.
Import Annotated, add_messages, operator.add
Default Without a reducer, updates overwrite the current key.
from typing import Annotated, TypedDict
import operator
from langgraph.graph import add_messages

class State(TypedDict):
    messages: Annotated[list, add_messages]
    search_results: Annotated[list[str], operator.add]
    user_id: str

def add_result(state: State):
    return {"search_results": ["doc-1"]}

# messages append intelligently;
# search_results list concatenates;
# user_id overwrites if returned.

Best: Add reducers for keys that multiple nodes or branches update in parallel.

Mistake: Let a list be overwritten when you intended append semantics, or use a non-deterministic reducer.

ND

Nodes

Work units that read state and return updates
Concept A node is a function or runnable that executes one step in the graph.
Use when Model call, tool call, retrieval, validation, transform, routing prep.
Signature node(state, config=None, runtime=None)
Return Dict partial state, Command, or interrupt flow.
from langchain_openai import ChatOpenAI

model = ChatOpenAI(model="gpt-4.1-mini")

async def call_model(state: State):
    response = await model.ainvoke(state["messages"])
    return {"messages": [response]}

def validate(state: State):
    if not state["messages"]:
        return {"errors": ["missing messages"]}
    return {}

builder.add_node("model", call_model)
builder.add_node("validate", validate)

Best: Keep nodes small, idempotent when possible, and aligned with the state schema.

Mistake: Put dangerous side effects in nodes without interrupt/approval controls.

ED

Edges & Routing

Control flow between nodes
Concept Edges decide the next node; conditional edges use routing functions.
Use when Branching, tool loop, approval flow, error path.
Constants START, END
Methods add_edge, add_conditional_edges
from typing import Literal
from langgraph.graph import START, END

def route(state: State) -> Literal["tools", "final"]:
    last = state["messages"][-1]
    return "tools" if last.tool_calls else "final"

builder.add_edge(START, "model")
builder.add_conditional_edges(
    "model",
    route,
    {"tools": "tools", "final": END},
)
builder.add_edge("tools", "model")

Best: Return clear route labels and type them with Literal so the graph is readable.

Mistake: Create loops without stop conditions or return route labels that do not map to nodes.

CS

Command & Send

Update state and control routing dynamically
Command A node can update state and choose goto in one return value.
Send Dynamic fan-out for map-reduce or parallel work.
Import from langgraph.types import Command, Send
Use when Dynamic routing, parallel per-item processing, subgraph navigation.
from typing import Literal
from langgraph.types import Command, Send

def decide(state: State) -> Command[Literal["research", "answer"]]:
    if state.get("needs_research"):
        return Command(update={"status": "researching"}, goto="research")
    return Command(goto="answer")

def fan_out(state: State):
    return [
        Send("process_doc", {"doc": doc})
        for doc in state["docs"]
    ]

builder.add_conditional_edges("split", fan_out)

Best: Use Command when state updates and routing belong to the same decision.

Mistake: Overuse dynamic routing when simple conditional edges would be easier to read.

MR

Map-Reduce (Parallel Orchestration)

Tận dụng Send để xử lý dữ liệu song song và gom cụm
from typing import Annotated
import operator
from langgraph.types import Send
from langgraph.graph import StateGraph, END

class MapReduceState(TypedDict):
    items: list[str]
    results: Annotated[list[str], operator.add]

def fan_out_items(state: MapReduceState):
    # Trả về danh sách tác vụ song song
    return [Send("process_item", {"item": item}) for item in state["items"]]

async def process_item(state: dict) -> dict:
    # Xử lý song song phi chặn (async)
    summary = await model.ainvoke(f"Summarize: {state['item']}")
    return {"results": [summary.content]}

builder = StateGraph(MapReduceState)
builder.add_node("process_item", process_item)
builder.add_conditional_edges(START, fan_out_items)
builder.add_edge("process_item", END)

ToolNode

Prebuilt node for executing tool calls
Concept Node runs tool calls from AIMessage and returns ToolMessages into state.
Use when ReAct/tool-calling agents where you do not want to parse tool calls manually.
Import from langgraph.prebuilt import ToolNode, tools_condition
Methods Use as a graph node: builder.add_node("tools", ToolNode(tools))
from langchain_core.tools import tool
from langgraph.prebuilt import ToolNode, tools_condition

@tool
def get_weather(city: str) -> str:
    """Get current weather."""
    return f"{city}: 28C"

tools = [get_weather]
model_with_tools = model.bind_tools(tools)

async def chatbot(state: MessagesState):
    res = await model_with_tools.ainvoke(state["messages"])

builder.add_node("chatbot", chatbot)
builder.add_node("tools", ToolNode(tools))
builder.add_conditional_edges("chatbot", tools_condition)
builder.add_edge("tools", "chatbot")

Best: Use clear tool schemas and control side effects with interrupts when needed.

Why: Execute tool calls correctly inside an agent loop and return tool results as messages.

Threads, checkpoints, resume, time travel
Concept Checkpointers store state snapshots by thread for resume and history inspection.
Use when Memory, human approval, fault tolerance, long-running agents.
Import InMemorySaver, Postgres/SQLite savers for production.
Config {"configurable": {"thread_id": "..."}}

config = {
    "configurable": {"thread_id": "conversation-42"}
}

# Production must use a durable checkpoint store.
# Production: Postgres checkpointer.
from langgraph.checkpoint.postgres import PostgresSaver

DB_URI = "postgresql://postgres:postgres@localhost:5432/ai?sslmode=disable"

with PostgresSaver.from_conn_string(DB_URI) as checkpointer:
    checkpointer.setup()  # run once during migration/bootstrap

    graph = builder.compile(checkpointer=checkpointer)
    config = {"configurable": {"thread_id": "support:ticket-8842"}}
# Async service: AsyncPostgresSaver + async graph runtime.
from langgraph.checkpoint.postgres.aio import AsyncPostgresSaver

async def build_async_graph():
    async with AsyncPostgresSaver.from_conn_string(DB_URI) as checkpointer:
        await checkpointer.setup()
        graph = builder.compile(checkpointer=checkpointer)

        config = {"configurable": {"thread_id": "chat:u-42"}}
        result = await graph.ainvoke(
            {"messages": [{"role": "user", "content": "Draft reply"}]},
            config=config,
        )

        async for update in graph.astream(
            {"messages": [{"role": "user", "content": "Revise it"}]},
            config=config,
            stream_mode="updates",
        ):
            print(update)
        return result

Best: Keep thread_id stable by conversation, run Postgres migrations deliberately, and use async savers in async servers.

Mistake: Use InMemorySaver for real production data, forget setup(), or mix random thread IDs with resumable workflows.

STO

Stores & Long-Term Memory

Cross-thread persistence for user facts
Concept Store keeps data outside a thread and accesses it by namespace/key.
Use when User profile, preferences, facts that must persist across conversations.
Import from langgraph.store.memory import InMemoryStore
Methods put, get, search, delete
from langgraph.store.memory import InMemoryStore
from langgraph.runtime import Runtime

store = InMemoryStore()
graph = builder.compile(store=store)

def save_preference(state, runtime: Runtime):
    user_id = runtime.context["user_id"]
    runtime.store.put(
        ("users", user_id),
        "preference",
        {"language": "vi"},
    )
    return {}

config = {"configurable": {"thread_id": "t-1"}}
context = {"user_id": "u-42"}
graph.invoke(input_state, config=config, context=context)

Best: Use checkpointers for thread state and Store for cross-thread user facts; namespace by tenant/user/product.

Mistake: Store uncontrolled PII, use Store as checkpoint replacement, or skip tenant isolation in SQL namespaces.

INT

Interrupts

Pause graph for human input or approval
Concept interrupt() pauses execution and resumes with Command(resume=...).
Use when Approval before dangerous tools, user input requests, state editing, and output review.
Import from langgraph.types import interrupt, Command
Requires Needs checkpointer to resume graph.
from langgraph.types import interrupt, Command

def approve_payment(state: State):
    answer = interrupt({
        "question": "Approve payment?",
        "amount": state["amount"],
    })
    if answer["approved"]:
        return {"approved": True}
    return {"approved": False}

graph = builder.compile(checkpointer=checkpointer)
config = {"configurable": {"thread_id": "pay-1"}}

# Execute async
async def run_payment():
    await graph.ainvoke({"amount": 100}, config)
    # Resume sau khi được duyệt
    await graph.ainvoke(Command(resume={"approved": True}), config)

Best: Interrupt before side effects and include enough payload for the UI to show the decision.

Mistake: Charge/send/delete before the interrupt, or omit an idempotency key for retried/resumed nodes.

STR

Streaming

Values, updates, messages, events, custom data
Concept Stream state updates, full values, LLM tokens, debug data, or custom events.
Use when Chat UI, progress display, tool-call visibility, observability.
Modes values, updates, messages, debug, custom
Methods stream, astream, astream_events
# Async streaming updates
async def run_stream():
    async for chunk in graph.astream(
        {"messages": [{"role": "user", "content": "Hi"}]},
        stream_mode="updates",
    ):
        print(chunk)

    # Async streaming LLM tokens
    async for token, metadata in graph.astream(
        input_state,
        stream_mode="messages",
    ):
        print(token.content, end="")

Best: use updates for progress, messages for token UI, custom for domain events, and events for tracing.

Mistake: Assume chunk formats are identical across stream modes.

IR

Interrupt Rules

Keep resume behavior deterministic and side effects safe
  • Before side effects
    Pause before payment, email, delete, ticket creation, or external write.
  • Stable order
    Do not reorder interrupt calls inside a node after checkpoints already exist.
  • No broad catch
    Do not swallow interrupt exceptions with a broad try/except Exception.
  • Idempotency
    External writes after resume need idempotency keys because nodes can be retried.

API

FastAPI + LangGraph Streaming API

FastAPI endpoint trả về LLM tokens stream theo thời gian thực
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
from pydantic import BaseModel
import json

app = FastAPI()

class ChatRequest(BaseModel):
    message: str
    thread_id: str

@app.post("/chat/stream")
async def chat_stream_endpoint(payload: ChatRequest):
    config = {"configurable": {"thread_id": payload.thread_id}}
    
    async def sse_generator():
        # stream_mode="messages" cho phép stream LLM tokens async
        async for chunk, metadata in graph.astream(
            {"messages": [("user", payload.message)]},
            config=config,
            stream_mode="messages"
        ):
            if chunk.content:
                yield f"data: {json.dumps({'token': chunk.content})}\n\n"
                
    return StreamingResponse(sse_generator(), media_type="text/event-stream")

RTE

Multi-Model Adaptive Routing

Định tuyến động: Model rẻ xử lý trước, nâng cấp lên model cao cấp khi cần
from langchain_openai import ChatOpenAI

async def route_by_complexity_node(state: MessagesState):
    cheap_model = ChatOpenAI(model="gpt-4.1-mini")
    res = await cheap_model.ainvoke(state["messages"])
    
    # Nếu câu trả lời chứa từ khóa không chắc chắn, chuyển hướng lên model o1
    if "unable to answer" in res.content.lower() or "complex query" in res.content.lower():
        premium_model = ChatOpenAI(model="gpt-4.1")
        upgrade_res = await premium_model.ainvoke(state["messages"])
        return {"messages": [upgrade_res]}
        
    return {"messages": [res]}
SUB

Subgraphs

Compose graphs inside graphs
Concept A compiled graph can be used as a node inside another graph.
Use when Multi-agent, reusable workflows, team/subtask isolation.
Shared State Parent and subgraph can share keys or transform input/output.
Methods Compile a subgraph, then add it with add_node("name", subgraph).
class SubState(TypedDict):
    question: str
    answer: str

sub = StateGraph(SubState)
sub.add_node("answer", lambda s: {"answer": "42"})
sub.add_edge(START, "answer")
sub.add_edge("answer", END)
subgraph = sub.compile()

class ParentState(TypedDict):
    question: str
    answer: str
    final: str

parent = StateGraph(ParentState)
parent.add_node("qa", subgraph)
parent.add_node("finalize", lambda s: {"final": s["answer"]})
parent.add_edge(START, "qa")
parent.add_edge("qa", "finalize")
parent.add_edge("finalize", END)

Best: Use subgraphs to package workflows with clear ownership and schema boundaries.

Mistake: Share too many state keys and create high parent/subgraph coupling.

SCH

Input/Output Schemas & Private State

Expose a clean public API while keeping internal orchestration state private
from typing import TypedDict
from langgraph.graph import StateGraph, START, END

class InputState(TypedDict):
    question: str

class OutputState(TypedDict):
    answer: str
    citations: list[str]

class InternalState(TypedDict):
    question: str
    rewritten_query: str
    retrieved_docs: list[str]
    critique: str
    answer: str
    citations: list[str]

def rewrite(state: InputState):
    return {"rewritten_query": state["question"].strip().lower()}

def retrieve(state: InternalState):
    return {"retrieved_docs": retriever.invoke(state["rewritten_query"])}

def answer(state: InternalState):
    return {
        "answer": model.invoke(format_prompt(state)).content,
        "citations": [doc.metadata["source"] for doc in state["retrieved_docs"]],
    }

builder = StateGraph(
    InternalState,
    input_schema=InputState,
    output_schema=OutputState,
)
builder.add_edge(START, "rewrite")
builder.add_node("rewrite", rewrite)
builder.add_node("retrieve", retrieve)
builder.add_node("answer", answer)
builder.add_edge("rewrite", "retrieve")
builder.add_edge("retrieve", "answer")
builder.add_edge("answer", END)

graph = builder.compile()
  • input_schema
    Public request contract. Keep it small and stable for APIs and UI clients.
  • output_schema
    Public response contract. Hide intermediate scratchpad, scores, and private tool state.
  • InternalState
    Full graph state with rewritten queries, retrieval results, critiques, approvals, and tool outputs.
  • Private channels
    Nodes can exchange internal keys that never appear in the final response.

SIG

Function Signatures

What each node receives, what to pass, and why
from typing import TypedDict
from langchain_openai import ChatOpenAI
from langgraph.graph import StateGraph, START, END
from langgraph.runtime import Runtime

class SupportContext(TypedDict):
    """Immutable per-run context.

    tenant_id: Which tenant owns this request.
    user_id: Authenticated user id for audit, retrieval filters, and store namespace.
    locale: UI/answer language preference.
    """
    tenant_id: str
    user_id: str
    locale: str

class SupportState(TypedDict):
    """Mutable graph state shared across nodes.

    question: User request entering the graph.
    context: Retrieved snippets safe to show the model.
    answer: Final answer returned by the graph.
    """
    question: str
    context: list[str]
    answer: str

model = ChatOpenAI(model="gpt-4.1-mini", temperature=0)
# vector_store: LangChain VectorStore created by your ingestion pipeline.
# It is captured by the node closure and filtered per tenant at runtime.

async def retrieve(state: SupportState, runtime: Runtime[SupportContext]) -> dict:
    """Node input: current state + runtime context.

    Pass tenant/user through runtime.context because it is request metadata,
    not mutable graph state. Return only changed state keys.
    """
    tenant_id = runtime.context["tenant_id"]
    tenant_retriever = vector_store.as_retriever(
        search_kwargs={"k": 4, "filter": {"tenant_id": tenant_id}}
    )
    docs = await tenant_retriever.ainvoke(state["question"])
    return {"context": [doc.page_content for doc in docs]}

async def answer(state: SupportState, runtime: Runtime[SupportContext]) -> dict:
    """Node output: partial state update.

    The graph merges {"answer": ...} into SupportState.
    """
    prompt = f"Answer in {runtime.context['locale']} from:\n{state['context']}"
    response = await model.ainvoke(prompt)
    return {"answer": response.content}

builder = StateGraph(SupportState, context_schema=SupportContext)
builder.add_node("retrieve", retrieve)
builder.add_node("answer", answer)
builder.add_edge(START, "retrieve")
builder.add_edge("retrieve", "answer")
builder.add_edge("answer", END)

RUN

Runtime, Config & Messages

Thread id, per-run context, message state, and stream metadata
State Mutable data the graph owns and checkpoints.
Context Immutable runtime data such as tenant, user, auth scope, locale.
Config Invocation metadata; configurable.thread_id selects checkpoint thread.
Messages Chat history usually lives under messages with add_messages reducer.
from typing import Annotated, TypedDict
from langchain_core.messages import AnyMessage
from langgraph.graph import StateGraph, add_messages
from langgraph.runtime import Runtime

class ChatState(TypedDict):
    """State persisted by the checkpointer.

    messages: Chat history; add_messages appends new messages instead of overwriting.
    route: Last router decision for debugging and tests.
    """
    messages: Annotated[list[AnyMessage], add_messages]
    route: str

class RunContext(TypedDict):
    """Runtime-only context.

    tenant_id: Used by retrievers/tools for filtering.
    plan: Billing or routing tier for model selection.
    """
    tenant_id: str
    plan: str

async def model_node(state: ChatState, runtime: Runtime[RunContext]) -> dict:
    """Read messages from state and tenant metadata from runtime.context."""
    selected_model = premium_model if runtime.context["plan"] == "pro" else basic_model
    response = await selected_model.ainvoke(state["messages"])
    return {"messages": [response], "route": runtime.context["plan"]}

CG

Compiled Graph Methods

How to call, stream, inspect, and mutate a compiled graph
  • invoke(input, config=None, context=None)
    Run synchronously and return the final state/output.
  • ainvoke(input, config=None, context=None)
    Async run for FastAPI, workers, and async model/retriever/tool calls.
  • stream(input, stream_mode=...)
    Sync iterator for values, updates, messages, debug, or custom chunks.
  • astream(input, stream_mode=...)
    Async stream for chat UIs and progress views.
  • astream_events(input, version="v3")
    Low-level event stream for tracing node starts/ends and custom events.
  • get_state(config)
    Read checkpoint snapshot for a thread; requires checkpointer.
  • update_state(config, values)
    Patch checkpointed state for admin fixes, human review, or time-travel workflows.
  • get_state_history(config)
    Inspect previous checkpoints to debug how state evolved.

FN

Functional API

Use @entrypoint and @task when a workflow reads cleaner than graph wiring
from langgraph.checkpoint.memory import InMemorySaver
from langgraph.func import entrypoint, task

checkpointer = InMemorySaver()

@task
async def retrieve_docs(question: str, tenant_id: str) -> list[str]:
    """Task input: question + tenant.

    Use @task for a resumable, retryable unit of work inside an entrypoint.
    """
    tenant_retriever = vector_store.as_retriever(
        search_kwargs={"filter": {"tenant_id": tenant_id}}
    )
    docs = await tenant_retriever.ainvoke(question)
    return [doc.page_content for doc in docs]

@entrypoint(checkpointer=checkpointer)
async def support_workflow(payload: dict) -> dict:
    """Entrypoint input: serializable request payload.

    payload["question"]: User question.
    payload["tenant_id"]: Access-control boundary for retrieval.
    """
    context = await retrieve_docs(payload["question"], payload["tenant_id"])
    return {"context": context}

result = await support_workflow.ainvoke(
    {"question": "Refund policy?", "tenant_id": "acme"},
    config={"configurable": {"thread_id": "support-fn-1"}},
)
KEY

Keywords

Core LangGraph terms and how they map to agent execution
  • StateGraph
    Graph builder where each node reads/writes typed shared state.
  • Node
    Function or runnable that returns partial state updates, commands, or interrupts.
  • Edge
    Static or conditional transition deciding the next node.
  • Reducer
    Merge rule for state keys updated by multiple nodes or parallel branches.
  • Command
    Return value that can update state and route with goto in one step.
  • Send
    Dynamic fan-out primitive for map-reduce and parallel branch creation.
  • Checkpointer
    Durable thread state store required for resume, time travel, and interrupts.
  • ToolNode
    Prebuilt LangGraph node that executes model tool calls and returns ToolMessages.
