LangGraph Overview

Low-level orchestration for stateful agents
Concept Framework for building agents and workflows as graphs with state, control flow, and persistence.
Use when Multi-step agents, human-in-the-loop flows, long-running workflows, and multi-agent systems.
Install pip install -U langgraph
Core APIs Graph API, Functional API, prebuilt agents, LangGraph Platform.
pip install -U langgraph langchain
pip install -U langchain-openai

# Main idea:
# StateGraph defines state.
# Nodes update state.
# Edges decide next node.
# Checkpointers persist execution.

Best: Use LangGraph when you need clearer state and control flow than a simple chain or agent.

Mistake: Build a custom graph when create_agent is already enough.

AG

Build An Agent

Prebuilt ReAct agent first, custom graph when control flow matters
from langchain.chat_models import init_chat_model
from langchain.tools import tool
from langgraph.checkpoint.memory import InMemorySaver
from langgraph.prebuilt import create_react_agent

@tool
def search_docs(query: str) -> str:
    """Use to search internal product documentation."""
    return "Relevant product documentation snippet."

@tool
def create_ticket(title: str, severity: str) -> str:
    """Use only after the user confirms ticket creation."""
    return f"ticket created: {title} / {severity}"

model = init_chat_model("openai:gpt-4.1", temperature=0)
checkpointer = InMemorySaver()

agent = create_react_agent(
    model=model,
    tools=[search_docs, create_ticket],
    checkpointer=checkpointer,
    prompt=(
        "You are a support agent. Search docs before answering. "
        "Ask for confirmation before creating tickets."
    ),
)

config = {"configurable": {"thread_id": "support-42"}}
result = agent.invoke(
    {"messages": [{"role": "user", "content": "How do I fix API timeout errors?"}]},
    config,
)

Start here: create_react_agent already gives you model calls, tool execution, message state, streaming, and checkpointing.

Go custom when: you need approval gates, deterministic routing, multi-agent teams, map-reduce, retries, or state editing.

LOOP

Custom Agent Loop

Model node -> tools node -> route until no tool calls
from typing import Literal
from langchain.chat_models import init_chat_model
from langchain.tools import tool
from langgraph.graph import MessagesState, StateGraph, START, END
from langgraph.prebuilt import ToolNode

@tool
def lookup_order(order_id: str) -> str:
    """Return order status."""
    return "shipped"

tools = [lookup_order]
model = init_chat_model("openai:gpt-4.1", temperature=0).bind_tools(tools)

def call_model(state: MessagesState):
    response = model.invoke(state["messages"])
    return {"messages": [response]}

def route_tools(state: MessagesState) -> Literal["tools", "__end__"]:
    last = state["messages"][-1]
    return "tools" if getattr(last, "tool_calls", None) else END

builder = StateGraph(MessagesState)
builder.add_node("agent", call_model)
builder.add_node("tools", ToolNode(tools))
builder.add_edge(START, "agent")
builder.add_conditional_edges("agent", route_tools)
builder.add_edge("tools", "agent")

graph = builder.compile()

Why this matters: this is the mental model behind tool-calling agents: call model, execute tools, append tool messages, call model again.

Do not hide risk: add interrupt() before side-effect tools such as payments, deletes, emails, or ticket creation.

SG

StateGraph

Main graph builder for stateful workflows
Concept Builder defines state schema, nodes, and edges, then compiles them into a runnable graph.
Use when Workflows with shared state and multiple coordinated steps.
Import from langgraph.graph import StateGraph, START, END
Methods add_node, add_edge, add_conditional_edges, compile
from typing import TypedDict
from langgraph.graph import StateGraph, START, END

class State(TypedDict):
    topic: str
    joke: str

def write_joke(state: State):
    return {"joke": f"Short joke about {state['topic']}"}

builder = StateGraph(State)
builder.add_node("write_joke", write_joke)
builder.add_edge(START, "write_joke")
builder.add_edge("write_joke", END)

graph = builder.compile()
result = graph.invoke({"topic": "graphs"})

Best: Type state clearly with TypedDict or Pydantic; nodes should return partial state updates.

Mistake: Mutate state directly or forget to compile the graph before invoking it.

ST

State & Reducers

Shared data and merge behavior
Concept State is the data each graph node reads and writes; reducers decide how updates merge.
Use when Append messages, aggregate results, parallel branches.
Import Annotated, add_messages, operator.add
Default Without a reducer, updates overwrite the current key.
from typing import Annotated, TypedDict
import operator
from langgraph.graph.message import add_messages

class State(TypedDict):
    messages: Annotated[list, add_messages]
    search_results: Annotated[list[str], operator.add]
    user_id: str

def add_result(state: State):
    return {"search_results": ["doc-1"]}

# messages append intelligently;
# search_results list concatenates;
# user_id overwrites if returned.

Best: Add reducers for keys that multiple nodes or branches update in parallel.

Mistake: Let a list be overwritten when you intended append semantics, or use a non-deterministic reducer.

ND

Nodes

Work units that read state and return updates
Concept A node is a function or runnable that executes one step in the graph.
Use when Model call, tool call, retrieval, validation, transform, routing prep.
Signature node(state, config=None, runtime=None)
Return Dict partial state, Command, or interrupt flow.
from langchain.chat_models import init_chat_model

model = init_chat_model("openai:gpt-4.1")

def call_model(state: State):
    response = model.invoke(state["messages"])
    return {"messages": [response]}

def validate(state: State):
    if not state["messages"]:
        return {"errors": ["missing messages"]}
    return {}

builder.add_node("model", call_model)
builder.add_node("validate", validate)

Best: Keep nodes small, idempotent when possible, and aligned with the state schema.

Mistake: Put dangerous side effects in nodes without interrupt/approval controls.

ED

Edges & Routing

Control flow between nodes
Concept Edges decide the next node; conditional edges use routing functions.
Use when Branching, tool loop, approval flow, error path.
Constants START, END
Methods add_edge, add_conditional_edges
from typing import Literal
from langgraph.graph import START, END

def route(state: State) -> Literal["tools", "final"]:
    last = state["messages"][-1]
    return "tools" if last.tool_calls else "final"

builder.add_edge(START, "model")
builder.add_conditional_edges(
    "model",
    route,
    {"tools": "tools", "final": END},
)
builder.add_edge("tools", "model")

Best: Return clear route labels and type them with Literal so the graph is readable.

Mistake: Create loops without stop conditions or return route labels that do not map to nodes.

CS

Command & Send

Update state and control routing dynamically
Command A node can update state and choose goto in one return value.
  • Send
    Dynamic fan-out for map-reduce or parallel work.
    • Import from langgraph.types import Command, Send
    Use when Dynamic routing, parallel per-item processing, subgraph navigation.
    from typing import Literal
from langgraph.types import Command, Send

def decide(state: State) -> Command[Literal["research", "answer"]]:
    if state.get("needs_research"):
        return Command(update={"status": "researching"}, goto="research")
    return Command(goto="answer")

def fan_out(state: State):
    return [
        Send("process_doc", {"doc": doc})
        for doc in state["docs"]
    ]

builder.add_conditional_edges("split", fan_out)

    Best: Use Command when state updates and routing belong to the same decision.

    Mistake: Overuse dynamic routing when simple conditional edges would be easier to read.

    MS

    MessagesState

    Convenient state schema for chat agents
    Concept Prebuilt state has a messages key with the add_messages reducer.
    Use when Chatbot, tool-calling agent, conversational workflow.
    Import from langgraph.graph import MessagesState
    Extend Subclass it to add keys such as user_id and summary.
    from langgraph.graph import MessagesState

class AgentState(MessagesState):
    user_id: str
    summary: str

def call_model(state: AgentState):
    response = model.invoke(state["messages"])
    return {"messages": [response]}

graph.invoke({
    "messages": [{"role": "user", "content": "Hi"}],
    "user_id": "u-42",
})

    Best: Use MessagesState for chat graphs instead of writing a message reducer yourself.

    Mistake: Overwrite all messages when you intended to append a new response.

    TN

    ToolNode

    Prebuilt node for executing tool calls
    Concept Node run tool calls from AIMessage and return ToolMessages into state.
    Use when ReAct/tool-calling agents where you do not want to parse tool calls manually.
    Import from langgraph.prebuilt import ToolNode, tools_condition
    Methods Use as a graph node: builder.add_node("tools", ToolNode(tools))
    from langchain.tools import tool
from langgraph.prebuilt import ToolNode, tools_condition

@tool
def get_weather(city: str) -> str:
    """Get current weather."""
    return f"{city}: 28C"

tools = [get_weather]
model_with_tools = model.bind_tools(tools)

def chatbot(state: MessagesState):
    return {"messages": [model_with_tools.invoke(state["messages"])]}

builder.add_node("chatbot", chatbot)
builder.add_node("tools", ToolNode(tools))
builder.add_conditional_edges("chatbot", tools_condition)
builder.add_edge("tools", "chatbot")

    Best: Use clear tool schemas and control side effects with interrupts when needed.

  • ToolNode
    Execute tool calls correctly inside an agent loop.
    • CP

    Persistence & Checkpointers

    Threads, checkpoints, resume, time travel
    Concept Checkpointers store state snapshots by thread for resume and history inspection.
    Use when Memory, human approval, fault tolerance, long-running agents.
    Import InMemorySaver, Postgres/SQLite savers for production.
    Config {"configurable": {"thread_id": "..."}}
    from langgraph.checkpoint.memory import InMemorySaver

checkpointer = InMemorySaver()
graph = builder.compile(checkpointer=checkpointer)

config = {
    "configurable": {"thread_id": "conversation-42"}
}

graph.invoke(
    {"messages": [{"role": "user", "content": "Remember me"}]},
    config,
)

state = graph.get_state(config)
history = list(graph.get_state_history(config))

# Production: use durable checkpoint store,
# e.g. PostgresSaver or SqliteSaver.

    Best: thread_id stable by conversation; use durable saver in production.

    Mistake: use InMemorySaver for real production data or not managing checkpoint growth.

    STO

    Stores & Long-Term Memory

    Cross-thread persistence for user facts
    Concept Store keeps data outside a thread and accesses it by namespace/key.
    Use when User profile, preferences, facts that must persist across conversations.
    Import from langgraph.store.memory import InMemoryStore
    Methods put, get, search, delete
    from langgraph.store.memory import InMemoryStore
from langgraph.runtime import Runtime

store = InMemoryStore()
graph = builder.compile(store=store)

def save_preference(state, runtime: Runtime):
    user_id = runtime.context["user_id"]
    runtime.store.put(
        ("users", user_id),
        "preference",
        {"language": "vi"},
    )
    return {}

config = {"configurable": {"thread_id": "t-1"}}
context = {"user_id": "u-42"}
graph.invoke(input_state, config=config, context=context)

    Best: Separate short-term thread state from long-term user memory.

    Mistake: Store uncontrolled PII or use Store as a replacement for checkpoint state.

    INT

    Interrupts

    Pause graph for human input or approval
    Concept interrupt() pauses execution and resumes with Command(resume=...).
    Use when Approval before dangerous tools, user input requests, state editing, and output review.
    Import from langgraph.types import interrupt, Command
    Requires Needs checkpointer to resume graph.
    from langgraph.types import interrupt, Command

def approve_payment(state: State):
    answer = interrupt({
        "question": "Approve payment?",
        "amount": state["amount"],
    })
    if answer["approved"]:
        return {"approved": True}
    return {"approved": False}

graph = builder.compile(checkpointer=checkpointer)
config = {"configurable": {"thread_id": "pay-1"}}

graph.invoke({"amount": 100}, config)
graph.invoke(Command(resume={"approved": True}), config)

    Best: Interrupt before side effects and include enough payload for the UI to show the decision.

    Mistake: Place interrupts inside non-idempotent nodes and then trigger side effects again on resume.

    STR

    Streaming

    Values, updates, messages, events, custom data
    Concept Stream state updates, full values, LLM tokens, debug data, or custom events.
    Use when Chat UI, progress display, tool-call visibility, observability.
    Modes values, updates, messages, debug, custom
    Methods stream, astream, astream_events
    for chunk in graph.stream(
    {"messages": [{"role": "user", "content": "Hi"}]},
    stream_mode="updates",
):
    print(chunk)

for token, metadata in graph.stream(
    input_state,
    stream_mode="messages",
):
    print(token.content, end="")

async for event in graph.astream_events(input_state):
    print(event["event"], event.get("name"))

    Best: use updates for progress, messages for token UI, events for tracing.

    Mistake: Assume chunk formats are identical across stream modes.

    SUB

    Subgraphs

    Compose graphs inside graphs
    Concept A compiled graph can be used as a node inside another graph.
    Use when Multi-agent, reusable workflows, team/subtask isolation.
    Shared State Parent and subgraph can share keys or transform input/output.
    Methods Compile a subgraph, then add it with add_node("name", subgraph).
    class SubState(TypedDict):
    question: str
    answer: str

sub = StateGraph(SubState)
sub.add_node("answer", lambda s: {"answer": "42"})
sub.add_edge(START, "answer")
sub.add_edge("answer", END)
subgraph = sub.compile()

class ParentState(TypedDict):
    question: str
    answer: str
    final: str

parent = StateGraph(ParentState)
parent.add_node("qa", subgraph)
parent.add_node("finalize", lambda s: {"final": s["answer"]})
parent.add_edge(START, "qa")
parent.add_edge("qa", "finalize")
parent.add_edge("finalize", END)

    Best: Use subgraphs to package workflows with clear ownership and schema boundaries.

    Mistake: Share too many state keys and create high parent/subgraph coupling.

    FN

    Functional API

    Workflow and task decorators
    Concept Write workflows with decorators instead of declaring a graph manually.
    Use when Flows that look like normal code but still need persistence and streaming.
    Import from langgraph.func import entrypoint, task
    Methods Call tasks with .result(); invoke or stream the workflow like a graph.
    from langgraph.func import entrypoint, task
from langgraph.checkpoint.memory import InMemorySaver

@task
def research(topic: str) -> str:
    return f"facts about {topic}"

@task
def write_answer(facts: str) -> str:
    return f"Answer: {facts}"

@entrypoint(checkpointer=InMemorySaver())
def workflow(topic: str) -> str:
    facts = research(topic).result()
    return write_answer(facts).result()

workflow.invoke("LangGraph")

    Best: Use the Functional API for linear or imperative workflows.

    Mistake: Use the Functional API when you need complex graph topology with many visually inspected routes.

    Concept Prebuilt agent graph for model + tools + state + checkpointer.
    Use when Fast ReAct/tool-calling agents that still use LangGraph state and persistence.

    Methods invoke, stream, and get_state when compiled with a checkpointer.
    from langgraph.prebuilt import create_react_agent
from langchain.chat_models import init_chat_model
from langchain.tools import tool

    """Search trusted docs."""
    return "LangGraph supports durable state."

model = init_chat_model("openai:gpt-4.1")
agent = create_react_agent(
    model=model,
    tools=[search],
    prompt="You are a concise assistant.",

    Mistake: Build a ReAct loop from scratch when a prebuilt agent already fits.

    MEM

    Memory Patterns

    Short-term, long-term, summarization
    Short-term Checkpointer + thread state keeps conversation history.
    Long-term Store stores facts/preferences across threads.
    Context Runtime context passes user_id, tenant_id, and request scope.
    Manage Trim/summarize messages to stay within the context window.
    from langgraph.graph import MessagesState

class ChatState(MessagesState):
    summary: str

def summarize_if_long(state: ChatState):
    messages = state["messages"]
    if len(messages) < 12:
        return {}
    summary = model.invoke([
        {"role": "system", "content": "Summarize chat."},
        *messages,
    ])
    return {
        "summary": summary.content,
        "messages": messages[-6:],
    }

    Best: Separate context memory from user knowledge memory.

    Mistake: Put the entire history into the prompt forever or store unverified memories.

    TT

    Time Travel & State Editing

    Inspect, fork, replay, update graph state
    Concept Use checkpoints to inspect history, resume from a checkpoint, and fork execution.
    Use when Debug agent, fix human input, replay workflow, audit path.
    Methods get_state, get_state_history, update_state
    Requires Graph compile with checkpointer.
    config = {"configurable": {"thread_id": "debug-1"}}

graph.invoke(input_state, config)

current = graph.get_state(config)
for checkpoint in graph.get_state_history(config):
    print(checkpoint.next, checkpoint.values)

graph.update_state(
    config,
    {"approved": True},
    as_node="human_review",
)

graph.invoke(None, config)

    Best: Use state history to debug routing and tool outputs instead of relying on print statements.

    Mistake: Update state from the wrong node/context and make replay hard to reason about.

    API

    Advanced API Map

    Core LangGraph functions, classes, and runtime operations
    • StateGraph(State)
      add_node, add_edge, add_conditional_edges, set_entry_point, set_finish_point, compile.
    • CompiledStateGraph
      invoke, ainvoke, stream, astream, get_state, get_state_history, update_state.
    • MessagesState
      Default chat state with messages and add_messages; extend it for user_id, summary, plan, or scratchpad.
    • Command
      Return Command(update=..., goto=...) for combined state update and routing, or Command(resume=...) after interrupts.
    • Send
      Dynamic fan-out for map-reduce, parallel workers, per-document analysis, or multi-agent delegation.
    • interrupt()
      Pause execution for human approval, user input, state edits, or risky tool calls.
    • ToolNode
      Executes tool calls from the last AI message and appends ToolMessages to state.
    • tools_condition
      Prebuilt router for model -> tools -> model loops when using message state.
    • Checkpointers
      InMemorySaver for demos; durable SQLite/Postgres savers for production threads and recovery.
    • Store
      put, get, search, delete for long-term memory outside a single thread.
    • @entrypoint / @task
      Functional API for imperative workflows that still need persistence, streaming, and retries.
    • LangGraph Platform
      langgraph.json, assistants, threads, runs, cron/background jobs, Studio, Server, and SDK clients.
    PRO

    Real-Life LangGraph Patterns

    Production orchestration patterns that show up outside toy demos
    • Support agent
      Retrieve docs -> ask clarification -> call diagnostic tools -> create ticket only after confirmation.
    • Research workflow
      Plan -> fan out searches with Send -> summarize each source -> synthesize -> cite -> review.
    • Data pipeline
      Extract -> validate -> route bad records -> human repair -> write approved records -> audit state.
    • Multi-agent team
      Supervisor graph routes to specialist subgraphs for research, coding, review, or customer support.
    • Approval workflow
      Use interrupt() before irreversible tools: payment, delete, email, deploy, account changes.
    • Long-running job
      Checkpoint every step; resume after failures; inspect state history for audit and debugging.
    OPS

    Failure Recovery & Reliability

    Design graphs assuming nodes, tools, and users will fail
    # Reliability checklist for every node
def node(state, config):
    # 1. Read only the state keys you need.
    # 2. Validate inputs before side effects.
    # 3. Make external writes idempotent.
    # 4. Return partial state updates only.
    # 5. Store enough metadata for replay/debug.
    return {"status": "validated"}

# Durable execution needs:
# - stable thread_id
# - production checkpointer
# - retry strategy for transient errors
# - interrupt before risky writes
# - state history inspection in support tools

    Best: Keep nodes small, deterministic where possible, and safe to re-run after checkpoint resume.

    Mistake: Put non-idempotent external writes before an interrupt or without a stable operation id.

    STATE

    State Design Best Practices

    Most graph bugs start with unclear state
    • Schema
      Use TypedDict or Pydantic; document which node owns each key.
    • Reducers
      Use reducers for append/aggregate keys; otherwise updates overwrite values.
    • Messages
      Use MessagesState for chat/tool loops; summarize or trim long histories.
    • Context
      Runtime user id, tenant, locale, and request metadata should be context/config, not graph state.
    • Audit
      Store decisions, tool ids, source ids, approval payloads, and failure reasons in state.
    • Boundaries
      Use subgraphs when teams, schemas, or responsibilities should be isolated.
    DEP

    Deployment Playbook

    Moving from notebook graph to maintained service
    • Graph factory
      Build graphs in a function; inject model, tools, checkpointer, store, and runtime settings.
    • Persistence
      Use durable checkpointers for production. In-memory savers are only for demos and tests.
    • Streaming UX
      Use updates for progress, messages for token UI, debug for internal tooling.
    • Observability
      Trace node names, state transitions, tool calls, interrupt payloads, latency, retries, and costs.
    • Testing
      Unit-test routing functions and nodes; integration-test full graph paths with fixed inputs.
    • Versioning
      Version graph topology, prompts, tools, state schema, and migration logic for old checkpoints.
    MAP

    Quick Decision Map

    Pick the right LangGraph component
    • StateGraph
      Workflow with multiple nodes, shared state, and explicit routing.
    • MessagesState
      Chat/tool-calling graph has message history.
    • Reducer
      Multiple nodes or branches update the same key, or you need append semantics.
    • Conditional edges
      Routing based on state after a node finishes.
    • Command
      A node updates state and chooses the next node.
    • Send
      Dynamic fan-out for map-reduce or parallel work.
    • Checkpointer
      Resume, memory by thread, interrupts, time travel.
    • Store
      Long-term memory across threads.
    • Interrupt
      Human approval/input before risky steps.
    • ToolNode
      Execute tool calls correctly inside an agent loop.
    • Functional API
      Imperative, linear workflows with little graph topology.
    • Subgraph
      Package sub-workflows, teams, or child agents with separate schemas.
